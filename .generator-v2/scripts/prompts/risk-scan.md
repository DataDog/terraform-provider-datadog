You are reviewing a Terraform data source that was generated by a code generator (tfgen)
from a slice of the Datadog v2 OpenAPI spec. It compiled and the generator reported
success. Your job has two parts:

1. Find MATERIAL runtime risks — behavior that can make this data source return wrong or
   empty results against a live org even though it builds.
2. Review the generated code for signs the generator got something WRONG or INCOMPLETE.
   The generator is good but not perfect and does not handle every schema shape. A clean
   build does not mean the mapping is right, so do not assume it is. If part of the code
   looks like it mishandled, dropped, or misrepresented something in the spec, flag it as a
   possible generator issue for a human to confirm.

You are given, after this prompt: the cardinality, the read/search operation ids, the flags
a script already recorded (do not repeat those), the generator report diagnostics, the
sliced OpenAPI (the source of truth), and the generated Go source.

## The boundary — review, don't re-implement

Do read the generated Go and compare it against the spec at a review level. Do NOT re-derive
the full correct mapping field by field, and do not rewrite from scratch what every field's
type or name "should" be. You are spotting credible problems, not reproducing the generator.
You do not need proof — a well-grounded "this looks wrong, a reviewer should check" is worth
flagging. Point at the specific field, type, or line that prompted it.

## Signs of a possible generator issue (review the code for these)

- A field in the response schema that has no matching schema attribute, or no assignment in
  the read/state-mapping — it may have been silently dropped.
- A field whose generated type does not seem to match the spec (for example a structured
  value represented as a plain string, or a collection represented as a single value).
- A nested or structured part of the response that the code seems to have collapsed,
  omitted, or represented in a way that would lose data.
- Read or state-mapping logic that looks incomplete or inconsistent with the response shape.
- A search/"both" path that does not enforce resolving exactly one match, or that selects a
  record through a path that cannot populate a field the schema exposes.
- Absence or 404 handling that maps a missing value to an empty string instead of null.

Only flag what the code and spec in front of you actually show. If the code looks like a
faithful, complete mapping, say so — do not invent problems.

## Material runtime risks (independent of code correctness)

### Plural (list/collection-backed)
- Returns the whole org with no server-side filter; the id is computed, so callers narrow
  client-side and tests must use set-membership, never a fixed index or a hard-coded count.
- Silent-empty trap: the SDK turns a strict-parse failure (unknown enum, missing required
  field, oversized integer) into an empty list with no error, so "empty" can mean "parse
  failed." Call this out when the schema has enums, required fields, or wide integers.
- Pagination: if the spec is paginated but the code does not paginate, results are truncated.
- Read-after-write lag: create-then-list in one apply can transiently return zero rows.

### Singular (by-id / "both" / search-only)
- "both" degradation: if the by-id detail type differs from the list item type, the search
  path implied by the inputs may not actually be wired.
- Detail-only fields (like a token) will be null if the source resolves via the list path.

## How to write reviewer notes

Each note is one or two concise sentences: what looks wrong or risky, why, and what to
check. Prefix a code-correctness finding with "possible generator issue:" so a reviewer can
tell it apart from a runtime risk. Ground every note in the spec or code you were given.

## Output

Return STRICT JSON and nothing else — no prose before or after, no markdown code fences:

{"material_risk": true or false, "risk_summary": "one line naming the top concern(s) for a callout at the top of the PR; empty string if none", "reviewer_notes": ["note", "note"]}

Set material_risk to true if any note is a genuine merge-blocking concern — including a
credible possible generator issue. If nothing material applies, return
{"material_risk": false, "risk_summary": "", "reviewer_notes": []}.
