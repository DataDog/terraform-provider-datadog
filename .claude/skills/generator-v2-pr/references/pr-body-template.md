# PR body template

Use this **exactly**. Populate the `<...>` slots. The footer line is verbatim — do not
reword it, and it is the only *development-status* text allowed in the PR.

## CI-gating metadata (the PR fails CI without all three)
- **Title**: `[<service>] Add datadog_<name> data source` — the `[<service>]` prefix is
  repo-enforced. Derive `<service>` from the endpoint's spec tag/annotation (e.g. `rum`).
- **Label**: `changelog/feature` (passed via `gh pr create --label`).
- **Verification disclaimer**: the first block of the body, below — required, not optional.

## Project-context disclaimer (the VERY FIRST thing in the body)
This block goes at the very top of the body, above everything else including the
verification disclaimer. Use it verbatim (it explains what this project is and sets
reviewer expectations):

```
> ℹ️ **This PR is part of a new project that auto-generates Terraform provider data
> sources to increase coverage.** The data source code is generated **deterministically
> by tfgen from an annotated OpenAPI spec, without the use of LLMs**. This PR is
> **reviewed via AI, though human review is still necessary**. If you decide to use
> this generated data source, please **review it thoroughly and test it** before relying
> on it — see the verification note and testing guide below.
```

## Required verification disclaimer (immediately after the project-context disclaimer)
This is a safety statement (separate from and in addition to the footer). It must be the
second block in the body, directly under the project-context disclaimer:

```
> ⚠️ **This PR contains auto-generated code and must be verified before merging.** Do not
> merge until the acceptance test has been recorded and replays green against the Frog org
> (see "How to test"). A clean build and a green generator report do **not** prove runtime
> correctness.
```

## Prominent risk callout (conditional)
If step 5 found any **material** risk, place this block immediately after the verification
disclaimer, above `### Generated`, so it's among the first things a reviewer reads. Omit
the block entirely if there are no material risks (don't write "no risks" theater — the
Reviewer notes section still lists minor items).

```
> ⚠️ **Merge risks flagged — read before approving.** <one-line summary of the top risk(s)>.
> Details under "Reviewer notes / risks" below.
```

## Docs callout (conditional — check whether `docs/data-sources/<name>.md` is in the change)
`make docs` / tfplugindocs generates `docs/data-sources/<name>.md`. Check the branch: if
the docs file **is** present, list it in `### Generated` (with `(created)`), no callout
needed. If it is **not** present, the CI docs check will fail — so add this **big-text**
action block immediately after the risk callout (above `### Generated`), and drop the docs
line from `### Generated`:

```
## 🚨 Run `make docs` before merge

`docs/data-sources/<name>.md` was not generated in this change. Run `make docs`, commit the
result, and push — CI's docs check fails without it.
```

## Filling notes
- **Status** values come straight from `artifacts[].status` (`created`/`updated`/`unchanged`).
- **Cassette line**: only write "recorded & replays green" if a cassette actually replayed
  green this run. Otherwise "scaffold, not yet recorded". Never overclaim (see SKILL.md top rule).
- **Registry vs hand-wire**: while generator-v2 is unmerged, keep the "Wired directly into
  `framework_provider.go`" line. Once generator-v2 merges and `datasources_generated.go`
  is the real wiring, swap this line accordingly.
- `docs/data-sources/<name>.md` — comes from `make docs` / tfplugindocs, not tfgen. Check
  the branch: if the docs file is present, list it in `### Generated` as `(created)`. If
  it's absent, drop the line and add the big-text "Run `make docs` before merge" callout
  (above `### Generated`) — CI's docs check fails without the docs file.
